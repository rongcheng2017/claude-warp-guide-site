用 WARP 落地解决 Claude app 的 Cloudflare 验证闪跳#

i一句话

Claude 桌面 app 在「正在进行安全验证」与「Couldn't connect to Claude」之间反复横跳,根因通常是机房 IP 信誉差,过不了 Cloudflare 的 JS challenge。在代理链路末端串一层 Cloudflare WARP 把出口 IP 洗成 WARP IP 可以解决 app 闪跳；但 WARP 只应该作为 Anthropic / Claude 定向落地修复层,不要全局代理。YouTube / Google Video 等流媒体继续走普通 PROXY 节点。 如果 Claude Code 仍报 403 Request not allowed,还要检查 Verge merge 脚本是否真的把 WARP / WARP-CLAUDE 合进 mihomo 运行配置。

适用环境#

本次实操环境,换环境时按此对照:

  • macOS(Apple Silicon)
  • 代理客户端:Clash Verge Rev 2.5.1,内核 Mihomo (verge-mihomo) v1.19.25
  • 混合端口:127.0.0.1:7897
  • 内核控制器:unix socket /tmp/verge/verge-mihomo.sock
  • 当前订阅:US-西海岸,落地节点 美国🇺🇸-西海岸-01(VLESS+Reality),选节点组名 PROXY
!前置条件

内核必须是 Mihomo / Clash.Meta——只有它支持 wireguard 出站 + dialer-proxy 链式代理。老版 Clash(Premium/原版)不行。

故障判断:怎么确认是这个问题#

bash 
# 1. 看当前出口 IP 与 Cloudflare 对它的处置
curl -s -x http://127.0.0.1:7897 --max-time 15 \
  https://claude.ai/cdn-cgi/trace | grep -iE "^(ip|warp)="

# 2. 看 claude.ai 是否被 challenge
curl -s -I -x http://127.0.0.1:7897 --max-time 20 \
  https://claude.ai/ | grep -iE "^(HTTP|cf-mitigated|server)"

判据:

  • 出口是机房 IP(如 ColoCrossing 的 107.174.x.x),warp=off
  • 响应头出现 cf-mitigated: challenge + HTTP 403
!curl 的 403 不代表 app 也进不去

cf-mitigated: **challenge**(区别于 block)是需要浏览器跑 JS 才能放行的挑战。curl 不执行 JS,所以永远停在 403——哪怕用最干净的住宅 IP 也是 403。 所以 curl 只能用来看出口 IP 和是否触发 challenge,不能用来判定 app 最终能否进。最终结论只能靠真实 app/浏览器实测。 机房 IP 闪跳的真正原因:IP 信誉太差,JS 跑完也不放行;或 IP 不稳定导致放行 cookie 频繁失效 → 反复横跳。

如果症状是 Claude Code 里出现:

tail -n 200 "$BASE/logs/latest.log" | rg -i 'WARP-CLAUDE|test failed|最小配置|proxy .*not found|配置验证'

命中下面这类日志,就是 Merge / Script 持久化没闭环:

这种场景直接跳到 3.5,不要继续反复 /login。

原理:链式代理 + WARP 落地#

text 
本地 Claude app → Clash(7897) → 你的VPS机房节点(PROXY) → WARP → claude.ai
                                                          ↑ 出口IP在这里被洗成 Cloudflare WARP IP

WARP 出口是 Cloudflare 自家 IP(如 104.28.x.x),Cloudflare 对它信任度高,JS challenge 能稳定通过。

!WARP 不是全局加速器

本方案只把 claude.ai / anthropic.com / claudeusercontent.com 这类 Anthropic / Claude 相关域名送进 WARP-CLAUDE。 其他站点,尤其 YouTube / Google Video,不要走 WARP。实测这类站点走 WARP 可能更慢甚至 timeout,应继续走普通 PROXY 节点或更换更适合流媒体的节点。

?为什么 API 通道(Claude Code 终端版)不受影响

api.anthropic.com 不走 claude.ai 那套浏览器人机验证。同一个被 challenge 的机房 IP 下,claude.ai 返回 403,而 api.anthropic.com 返回 401(已到后端,只是没带 key)。 所以这道题如果只是想用 Claude 干活,Claude Code 终端版通常能绕开 challenge。 但如果 Claude Code 自己也报 API Error: 403 Request not allowed,先不要判断账号坏了,要按 3.5 检查 Clash Verge 是否回退成最小配置。

完整执行步骤#

1. 安装 wgcf 并生成 WARP 配置(匿名,免费)#

bash 
brew install wgcf

mkdir -p ~/.config/warp && cd ~/.config/warp
# 注册走现有代理出去
export HTTPS_PROXY=http://127.0.0.1:7897 HTTP_PROXY=http://127.0.0.1:7897 ALL_PROXY=http://127.0.0.1:7897
wgcf register --accept-tos      # 生成 wgcf-account.toml
wgcf generate                   # 生成 wgcf-profile.conf

产物 ~/.config/warp/wgcf-profile.conf 含字段:PrivateKey、Address、PublicKey(Cloudflare 固定 bmXOC+F1FxEMF9dyiK2H5/1SUtzH0JuVo51h2wPfgyo=)、Endpoint。

!提取私钥不要用 sed 's/.*= *//'

base64 私钥末尾的 = 会被贪婪匹配吃掉。用 awk 按分隔符切:

bash 
PK=$(awk -F' = ' '/^PrivateKey/{print $2}' ~/.config/warp/wgcf-profile.conf)
# 正常长度 44

2. 临时验证(注入运行时配置 + 热重载)#

直接改运行时 clash-verge.yaml 只为快速验证。一旦关代理 / 更新订阅 / 重启 Verge,内核会从订阅重新生成配置,WARP 块就被冲掉(实操中踩过这个坑:关代理重开后 app 又开始闪)。验证通过后必须做第 3 步持久化。

运行时配置路径:

3. 持久化(关键,防订阅更新 / 关代理覆盖)#

把 WARP 写进当前订阅绑定的 Merge 扩展配置文件。Verge 每次生成配置都会自动合入,不再丢。

定位 Merge 文件:看 profiles.yaml 中 current: 指向的订阅,其 option.merge 字段的 uid 即对应 profiles/<uid>.yaml。

  • 本例:订阅 US-西海岸 → merge uid mRW52hstmzq9 → 文件 profiles/mRW52hstmzq9.yaml

在该 Merge 文件中追加(Verge 合并语法):

yaml 
# prepend-rules 顶部加分流(若已有 prepend-rules 段就并进去)
prepend-rules:
  # WARP 只作为 Anthropic / Claude 定向落地修复层。
  # 流媒体和日常浏览不要走 WARP,继续走 PROXY。
  - DOMAIN-SUFFIX,claude.ai,WARP-CLAUDE
  - DOMAIN-SUFFIX,anthropic.com,WARP-CLAUDE
  - DOMAIN-SUFFIX,claudeusercontent.com,WARP-CLAUDE

  # 如果你的 Merge 里有 YouTube / Google Video 规则,保持 PROXY,不要改成 WARP。
  - DOMAIN-KEYWORD,youtube,PROXY
  - DOMAIN-SUFFIX,googlevideo.com,PROXY

append-proxies:
  - name: WARP
    type: wireguard
    server: 162.159.192.1
    port: 2408
    ip: 172.16.0.2
    ipv6: 2606:4700:110:8560:f32c:59c1:df01:6587   # 用 wgcf-profile.conf 里的 Address
    private-key: <从 ~/.config/warp/wgcf-profile.conf 的 PrivateKey 填入>
    public-key: bmXOC+F1FxEMF9dyiK2H5/1SUtzH0JuVo51h2wPfgyo=
    udp: true
    mtu: 1280
    dialer-proxy: PROXY        # 指向选节点组名,不写死节点名

append-proxy-groups:
  - name: WARP-CLAUDE
    type: select
    proxies:
      - WARP
      - PROXY                  # 可随时切回直连主节点对比
!dialer-proxy 指向组名而非节点名

写 dialer-proxy: PROXY(选节点组),不要写死 美国🇺🇸-西海岸-01。订阅更新后节点名可能变(-01→-02),组名 PROXY 稳定,更抗订阅变动。

3.5. 修复 Verge Merge 脚本,防 WARP-CLAUDE 丢失#

!另一个坑:Merge 写了,但 mihomo 还是没收到

Clash Verge 的 Merge 扩展可以写 prepend-rules、append-proxies、append-proxy-groups。但这些字段不是 mihomo 原生字段。如果当前 profile 绑定的 Script 只把 prepend-rules 合进 rules,就会出现规则已经引用 WARP-CLAUDE,代理组却没合进去的半残配置。

典型报错:

text

configuration file .../clash-verge-check.yaml test failed

然后 Verge 会回退成默认最小配置。此时 127.0.0.1:7897 端口可能还通,但运行内核只有 DIRECT / REJECT / GLOBAL,没有订阅节点、没有 WARP、没有 Claude 规则。Claude Code 里看到的 API Error: 403 Request not allowed 可能就是这个下游症状。

把脚本补成同时处理 rules、proxies、proxy-groups:

javascript 
function main(config, profileName) {
  if (!config || typeof config !== "object") return config;

  mergeNamedList(config, "proxies", "prepend-proxies", "append-proxies");
  mergeNamedList(config, "proxy-groups", "prepend-proxy-groups", "append-proxy-groups");
  mergeValueList(config, "rules", "prepend-rules", "append-rules");

  return config;
}

function mergeNamedList(config, targetKey, prependKey, appendKey) {
  mergeList(config, targetKey, prependKey, appendKey, function(item) {
    if (item && typeof item === "object" && item.name) return item.name;
    return stringifyKey(item);
  });
}

function mergeValueList(config, targetKey, prependKey, appendKey) {
  mergeList(config, targetKey, prependKey, appendKey, stringifyKey);
}

function mergeList(config, targetKey, prependKey, appendKey, keyForItem) {
  var prepend = Array.isArray(config[prependKey]) ? config[prependKey] : [];
  var existing = Array.isArray(config[targetKey]) ? config[targetKey] : [];
  var append = Array.isArray(config[appendKey]) ? config[appendKey] : [];

  delete config[prependKey];
  delete config[appendKey];

  if (prepend.length === 0 && append.length === 0) return;

  var seen = {};
  var merged = [];

  function pushItem(item) {
    if (item == null) return;
    var key = keyForItem(item);
    if (seen[key]) return;
    seen[key] = true;
    merged.push(item);
  }

  for (var i = 0; i < prepend.length; i++) pushItem(prepend[i]);
  for (var j = 0; j < existing.length; j++) pushItem(existing[j]);
  for (var k = 0; k < append.length; k++) pushItem(append[k]);

  config[targetKey] = merged;
}

function stringifyKey(item) {
  return typeof item === "string" ? item : JSON.stringify(item);
}

无效 / 别浪费时间的方向#

  • 改 UA、改系统时间、清缓存:对基于 IP 信誉的 challenge 几乎无效(时间正常时尤其)。
  • 反复点 Refresh:IP 不变,结果不变。
  • 浏览器手动过验证再喂 cookie 给 app:app 是独立 Electron webview,cookie 不与系统浏览器共享,不可行。

关键认知沉淀#

  1. cf-mitigated: challenge 要浏览器跑 JS,curl 永远 403,curl 结论 ≠ app 结论。
  2. 机房 IP 是原罪,WARP 把出口洗成 Cloudflare 自家 IP 来过 challenge。
  3. 临时注入运行时配置必被订阅重生成覆盖,持久化要写进 Merge 扩展配置。
  4. Merge 文件和 Script 文件是一组:Merge 负责声明 append-*,Script 负责把它们折叠成 mihomo 原生字段。
  5. WARP 只给 Anthropic / Claude 定向使用,不要全局套 WARP;YouTube / Google Video 这类流媒体应走普通 PROXY 或专门的流媒体节点。
  6. API 通道(Claude Code 终端版)通常绕过 claude.ai 登录页 challenge,但 Claude Code 自己报 403 时,先查运行内核是否回退到了最小配置。
